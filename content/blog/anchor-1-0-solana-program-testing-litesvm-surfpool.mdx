Anchor's first stable major release shipped in April 2026. If you have been writing Solana programs against Anchor 0.31 (or earlier), `anchor build` and `anchor test` still feel familiar — but underneath, the testing pipeline has been rebuilt almost end to end. The default local validator is different. The default test template is different. The TypeScript client lives at a new package name. And Anchor no longer depends on the Solana CLI being installed on your PATH.

This post walks through what actually changed, why the new defaults matter, and what you need to update in an existing project.

## Why Anchor 1.0 Restructures the Solana Program Testing Toolchain

The pain points behind 1.0 are familiar to anyone who has iterated on a Solana program at scale. `solana-test-validator` boots in seconds, not milliseconds, and tearing it down between runs adds up across CI workers. The old `@coral-xyz/anchor` TypeScript package was tightly coupled to legacy `@solana/web3.js` v1, so adopting Solana Kit (the new name for web3.js v2) meant juggling two import graphs. And requiring the Solana CLI binary on PATH made Anchor harder to ship into hermetic CI environments.

Anchor 1.0 addresses each of those independently. The CLI now ships native implementations of `balance`, `airdrop`, `address`, and `deploy`, so a separate `solana` install is no longer required for typical workflows. Surfpool replaces `solana-test-validator` as the default validator for `anchor test` and `anchor localnet`. LiteSVM becomes the default test template generated by `anchor init`. And the TypeScript packages have moved to a new namespace — `@anchor-lang/core` — to keep the boundary with the legacy client cleaner.

## LiteSVM as the Default Test Template

LiteSVM is an in-process Solana VM. There is no validator binary, no RPC server, no genesis bootstrap. You instantiate a struct, load your compiled program, and execute instructions directly against the SVM in the same process as your test. Startup is on the order of milliseconds, and tearing down is just dropping the struct.

A minimal LiteSVM test in Rust — close to what `anchor init` now scaffolds for you:

```rust
use litesvm::LiteSVM;
use solana_keypair::Keypair;
use solana_signer::Signer;
use solana_message::Message;
use solana_transaction::Transaction;

let mut svm = LiteSVM::new();
let payer = Keypair::new();
svm.airdrop(&payer.pubkey(), 1_000_000_000).unwrap();

let blockhash = svm.latest_blockhash();
let msg = Message::new_with_blockhash(&[ix], Some(&payer.pubkey()), &blockhash);
let tx = Transaction::new(&[&payer], msg, blockhash);
let meta = svm.send_transaction(tx).unwrap();
```

`svm.airdrop()` funds the payer; `svm.send_transaction()` executes a transaction and returns metadata you can assert against. There is also `add_program()` to load arbitrary `.so` bytes, `get_account()` to inspect post-execution state, and `set_sysvar::<Clock>()` to manipulate time when you need to test time-dependent logic without waiting for real slots. LiteSVM is also available as a TypeScript package (`npm i litesvm`) and inside the Python `solders` library if your test stack already lives in those languages.

The tradeoff is what LiteSVM does not simulate: networking, slot-by-slot scheduling, fee market behavior, or anything that depends on a live validator gossip layer. Use it for instruction logic, account state, and CPI correctness. Reach for Surfpool when you need realistic validator behavior.

## Surfpool as the New Local Validator

For everything LiteSVM intentionally skips, Anchor 1.0 boots Surfpool by default for `anchor test` and `anchor localnet`. Surfpool's pitch is faster startup than `solana-test-validator` while still giving you mainnet fork capabilities and a real RPC endpoint. If your existing test rig depends on the older binary — for example, CI scripts that hit `solana-test-validator` directly, or fixtures that assume a specific genesis — you can opt out for now:

```bash
anchor test --validator legacy
```

That is a transition flag, not a permanent escape hatch. New projects should plan around Surfpool being the path forward.

## Migrating an Existing Anchor Project

The minimum to update an existing project to 1.0:

1. Update `Cargo.toml` and `Anchor.toml` to the 1.0 versions and run `avm use 1.0.0`. AVM has its own `avm self-update` command now, with a passive warning when your installed version drifts.
2. In your TypeScript client, replace `import * as anchor from "@coral-xyz/anchor"` with `import * as anchor from "@anchor-lang/core"`. The new package is currently still pinned to the legacy `@solana/web3.js` v1 line, so do not bundle it alongside Solana Kit without isolating the import graphs.
3. If Surfpool is not installed locally, add `--validator legacy` to your test script while you migrate, or install Surfpool and adopt the new default.
4. Optionally adopt LiteSVM for any new test files. Anchor 1.0 generates them by default, but existing Mocha suites running against `solana-test-validator` keep working.

Two further additions are worth knowing about even if they do not drive your migration. Lifecycle hooks (`pre-build`, `post-build`, `pre-test`, `post-test`, `pre-deploy`, `post-deploy`) are now configurable in `Anchor.toml`, which removes a lot of the glue scripting that wrapped `anchor build` and `anchor deploy` in CI. And `Migration<'info, From, To>` is a new account type that lets you express schema migrations across program versions with the type system enforcing the transition rather than your own ad-hoc reinitialization code.

## Where Program Tests Stop and dApp Tests Begin

LiteSVM and Surfpool both target program-level testing — instruction execution, account state, CPI behavior, time-dependent logic via sysvar manipulation. None of them replace browser-level dApp coverage. Once your program is verified at the SVM level, the wallet connection prompt, the transaction approval popup, and the async confirmation UI are still where most users actually fall off. `@avalix/chroma` covers that browser layer today for MetaMask (EVM) and Polkadot.js / Talisman (Substrate) wallets, with Phantom and SVM dApp flows handled outside the library at the moment. The point is that SVM-level tests and dApp E2E tests answer different questions, and Anchor 1.0 only changes the first one.

The actionable upgrade path is short: bump `avm`, change the import path on the TS client, and decide whether your suite stays on `--validator legacy` or moves to LiteSVM and Surfpool. The defaults are what changed; the programs you have already written compile and behave the same.
